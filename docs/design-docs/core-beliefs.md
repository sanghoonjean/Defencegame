# Core Beliefs — MakeDefence

이 문서는 MakeDefence 개발 중 어떤 결정을 내려야 할 때
기준점이 되는 핵심 철학을 정의한다.
새로운 기능을 추가하거나 기존 시스템을 변경할 때
반드시 이 원칙에 부합하는지 먼저 확인한다.

---

## 1. 빌드 실험이 핵심 재미다

플레이어는 "이 조합이 과연 통할까?" 라는 질문을 스스로 던지고,
직접 실험해서 답을 찾는 과정에서 재미를 느껴야 한다.

**이 원칙이 의미하는 것:**
- 아이템 옵션 조합의 경우의 수는 충분히 많아야 한다
- 정답 빌드가 하나로 수렴되면 안 된다 — 다양한 빌드가 각자의 방식으로 유효해야 한다
- 큐브 시스템은 실험을 장려해야 한다 — 리롤, 추가, 제거가 부담 없이 시도될 수 있어야 한다

**이 원칙에 반하는 것:**
- 특정 타워 + 특정 옵션 조합이 압도적으로 강해 다른 선택지가 의미 없어지는 상황
- 아이템 옵션이 너무 단순해서 고민할 여지가 없는 상황

---

## 2. 성장감은 눈에 보여야 한다

웨이브를 거듭할수록 타워가 강해지고, 이전엔 위협적이었던 적이
이제는 쉽게 처리되는 경험이 명확하게 느껴져야 한다.

**이 원칙이 의미하는 것:**
- 아이템 옵션이 쌓일수록 타워의 성능 변화가 수치와 시각적으로 명확해야 한다
- 초반 웨이브와 후반 웨이브의 체감 난이도 차이가 빌드 성장으로 역전될 수 있어야 한다
- 복제형 큐브, 최상위 큐브 같은 희귀 자원 사용 시 확실한 파워 스파이크가 있어야 한다

**이 원칙에 반하는 것:**
- 열심히 크래프팅했는데 체감 변화가 없는 상황
- 성장이 수치로만 존재하고 플레이 경험상 차이가 없는 상황

---

## 3. 실패는 좌절이 아니라 학습이다

라스트 보스 도전에 실패했을 때, 플레이어는 "왜 실패했는지"를
명확히 알 수 있어야 하고 "다음엔 어떻게 해볼까"가 자연스럽게 떠올라야 한다.

**이 원칙이 의미하는 것:**
- 실패 원인이 불분명한 랜덤 요소에 의한 것이어선 안 된다
- 보스 패턴, 적 스탯은 충분한 정보가 플레이어에게 공개되어야 한다
- 실패 후 이전 빌드 상태를 되돌아볼 수 있어야 한다

**이 원칙에 반하는 것:**
- 이유를 알 수 없는 갑작스러운 패배
- 실패 시 진행 상황이 전부 초기화되어 다시 시도할 의욕이 사라지는 구조

---

## 4. 플레이어가 페이스를 결정한다

빌드를 고민하고 실험하는 시간은 플레이어가 스스로 통제해야 한다.
게임이 플레이어를 압박해서 판단을 서두르게 만들면 안 된다.

**이 원칙이 의미하는 것:**
- 웨이브는 플레이어가 준비됐을 때 시작한다 (수동 생성 기본)
- 자동 웨이브는 플레이어가 원할 때만 활성화하는 선택지다
- 인벤토리 조작, 큐브 사용은 웨이브 외 언제든 가능해야 한다

**이 원칙에 반하는 것:**
- 강제 타이머로 웨이브가 자동 시작되는 구조 (자동 토글 비활성화 시)
- 웨이브 중 인벤토리/큐브 사용이 불가능한 구조

---

## 5. 복잡함은 숨기지 않는다

MakeDefence는 PoE 스타일의 깊이 있는 시스템을 지향한다.
처음엔 어렵게 느껴져도 괜찮다 — 익힐수록 강력한 빌드가 보이는 구조가 목표다.

**이 원칙이 의미하는 것:**
- 아이템 옵션 테이블, 큐브 효과는 플레이어에게 완전히 공개된다
- 확률과 가중치는 숨기지 않고 UI에 표시한다
- 튜토리얼은 시스템을 단순화하지 않고 그대로 설명한다

**이 원칙에 반하는 것:**
- 복잡한 시스템을 감추기 위해 정보를 제한하는 것
- 깊이를 희생하고 단순화하는 것

---

## 요약

| 원칙 | 한 줄 요약 |
|------|-----------|
| 빌드 실험 | 다양한 조합이 각자의 방식으로 유효해야 한다 |
| 성장감 | 강해지는 느낌이 눈에 보여야 한다 |
| 실패 = 학습 | 실패 후 "다음엔 이렇게" 가 자연스럽게 떠올라야 한다 |
| 플레이어 페이스 | 게임이 플레이어를 압박하지 않는다 |
| 복잡함 공개 | PoE처럼 깊이를 숨기지 않는다 |
